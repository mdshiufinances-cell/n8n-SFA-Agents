# Bessie — n8n Workflow Design

**Version:** 2.4
**Last Updated:** 2026-06-19
**Status:** Chat Agent workflow built and live. Tool 4 (Read Yardi Files) production-tested. Tool 5 (Generate Excel) built but NOT yet tested end-to-end.

## Overview

One agent (Bessie) accessible two ways:

| Entry Point | Purpose | Status |
|---|---|---|
| Chat Trigger (n8n Chat Hub) | Always-on Q&A + on-demand forecasting | ✅ Live |
| Manual/Scheduled Forecast Run | Batch quarterly forecast across all properties | ⏳ Not yet built (originally planned, deferred — chat-driven forecasting via tools is the current approach instead) |

The original design assumed two separate workflows (Chat Agent + a separate batch Forecast Run workflow). In practice, **the Chat Agent workflow with the `generate_excel` tool covers the forecasting use case on-demand**, so the separate batch Forecast Run workflow has not been built and may not be needed. Revisit only if quarterly batch automation (no human in the loop) becomes a requirement.

## Tool 4 — Read Yardi Files: FINAL Working Architecture (v2.4)

**Status: ✅ Production-tested for prop01 (found) and prop02 (not found)**

### Final Node Structure

[When Executed by Another Workflow]
        ↓
[Search files and folders]   (Always Output Data: ON)
        ↓
[Match File]                  (Code node, Always Output Data: ON)
        ↓
[IF: found === true]
   ↙ true                  ↘ false
[Download file]      [Not Found Message]  (Code node, final node)
   ↓
[Extract from File]   (Always Output Data: ON)
   ↓
[Code in JavaScript]   (formats success output)

### Trigger Input Schema (case-sensitive!)

Field names as defined in "When Executed by Another Workflow":

PropertyCode  (String)
FileType      (String)

⚠️ IMPORTANT: n8n expressions are case-sensitive. All downstream references MUST use the exact casing: $json.PropertyCode, not $json.propertyCode. This caused a multi-hour outage when the Search node's query still referenced the lowercase version after the dynamic expression was reinstated.

### Search files and folders config
| Parameter | Value |
|---|---|
| Search Query | {{ $json.PropertyCode }} |
| Filter | Folder = yardi-input |
| Always Output Data | ON |

### Match File (Code node) — replaces old "Filter" node approach

    const triggerData = $('When Executed by Another Workflow').first().json;
    const fileType = triggerData.FileType;
    const propertyCode = triggerData.PropertyCode;

    const items = $input.all();

    const patternMap = {
      actuals: 'actualbudget',
      budget: 'monthbudget',
      rent_roll: 'rentroll'
    };
    const pattern = patternMap[fileType] || fileType;

    const match = items.find(item => {
      const name = (item.json.name || '').toLowerCase().replace(/_/g, '');
      return item.json.id && name.includes(pattern);
    });

    if (match) {
      return [{ json: { found: true, id: match.json.id, name: match.json.name, propertyCode, fileType } }];
    }
    return [{ json: { found: false, propertyCode, fileType } }];

### IF node
| Parameter | Value |
|---|---|
| Condition | {{ $json.found }} is true (Boolean) |

### Not Found Message (Code node, FALSE branch, final node)

    const j = $input.first().json;
    return [{
      json: {
        output: JSON.stringify({
          success: false,
          message: `No ${j.fileType} file found for property ${j.propertyCode}. Please confirm the file has been uploaded to /yardi-input/ with the correct naming convention.`
        })
      }
    }];

### Code in JavaScript (TRUE branch, final node after Extract from File)

    const items = $input.all();

    if (!items || items.length === 0 || !items[0].json || Object.keys(items[0].json).length === 0) {
      const propertyCode = $('When Executed by Another Workflow').first().json.PropertyCode;
      const fileType = $('When Executed by Another Workflow').first().json.FileType;
      return [{
        json: {
          output: JSON.stringify({
            success: false,
            message: `No ${fileType} file found for property ${propertyCode}.`
          })
        }
      }];
    }

    const fileType = $('When Executed by Another Workflow').first().json.FileType || 'actuals';
    const rows = items.map(item => item.json);

    const keyGLCodes = ['41100','41110','41200','41990','42100','42990',
      '43100','43290','43510','44100','44990','45990',
      '61050','62000','63000','64000','65000','66000','69990'];

    let outputRows;
    if (fileType === 'actuals' || fileType === 'budget') {
      const filtered = rows.filter(row => {
        const gl = String(Object.values(row)[0] || '').trim();
        return keyGLCodes.includes(gl) || gl.endsWith('990') || gl.endsWith('000');
      });
      outputRows = filtered.length > 0 ? filtered : rows.slice(0, 100);
    } else {
      outputRows = rows.slice(0, 200);
    }

    return [{
      json: {
        output: JSON.stringify({ success: true, fileType, rowCount: outputRows.length, data: outputRows })
      }
    }];

### Trigger Input Schema

⚠️ Check exact field casing matches what's referenced inside the Code node (PropertyCode/property — confirm which was actually used when this was built).

### How forecastData is populated

Tool 5 does NOT receive Tool 4's raw output directly. The flow is:
1. Bessie calls Tool 4 (read_files) → gets filtered GL row data back
2. Bessie reasons over that data using the calculator tool — computes baseline/upside/downside scenarios, variance, assumptions
3. Bessie constructs a NEW JSON object matching the Excel schema:

    {
      "property": "...",
      "period": "...",
      "actuals": [...],
      "baseline": [...],
      "upside": [...],
      "downside": [...],
      "variance": [...],
      "assumptions": [...]
    }

4. Bessie calls Tool 5 (generate_excel) with that constructed object as forecastData

### Suspected issues, based on Tool 4 lessons (UNVERIFIED — confirm before fixing)

1. **Respond to Webhook node** — likely wrong, same as Tool 4's original bug. "When Executed by Another Workflow" triggers should end with a node that returns a field named output as a string (via JSON.stringify), not a Respond to Webhook node. Recommend deleting it and confirming the Google Drive Upload node (or a Code node after it) is the true final node, formatted correctly.
2. **Always Output Data** — unconfirmed whether this is enabled on the Code/Upload nodes. Should be ON throughout, per Tool 4 lesson #3.
3. **Field casing** — unconfirmed whether property/period/forecastData casing is consistently referenced. Audit before testing.
4. **No graceful failure path** — Tool 4 needed an explicit found/not-found branch. Tool 5 has no equivalent (e.g., what happens if forecastData is malformed JSON, or the Google Drive upload fails). Consider adding a try/catch in the Code node and a fallback output message.

### Excel Generator Code (code/excel-generator.js)

See /code/excel-generator.js in repo — builds 6 tabs (Actuals YTD, Baseline Forecast, Upside, Downside, Budget vs Forecast, Assumptions) using the xlsx package. This part of the logic has not changed since originally written and is believed correct; the risk is entirely in the trigger/output wiring around it, not the Excel-building logic itself.

## Key Lessons Learned (read before modifying ANY sub-workflow)

1. **n8n field names are case-sensitive.** Always match the exact casing defined in the trigger's Workflow Input Schema.
2. **ToolWorkflow v2 requires the final node to return a field literally named `output`, as a STRING** (use JSON.stringify). Returning a raw JSON object causes "workflow did not return a response" errors.
3. **"Always Output Data" must be enabled on every node in the chain** (Search, Match File, Download, Extract, Code) to prevent the workflow from silently halting when zero items are found upstream — without it, n8n stops execution and returns nothing rather than a graceful error.
4. **Large P&L files exceed LLM context limits.** The full GL detail file is ~217K tokens — over Claude's 200K limit. The Code node filters to only key GL codes and subtotal rows (ending in 990/000) before returning data to Bessie.
5. **Avoid "Filter" node + downstream nodes for found/not-found logic.** Use an explicit Code node (to evaluate a found boolean) + an IF node branch instead. The Filter node's "Kept: 0 items" behaviour doesn't propagate usable data downstream for the false case.
6. **"Respond to Webhook" is the wrong final node for "When Executed by Another Workflow" sub-workflows.** This is a common mistake — Respond to Webhook pairs with Webhook triggers, not Execute Workflow triggers. Just let the last node's output flow back automatically (formatted with the output field per lesson #2).
